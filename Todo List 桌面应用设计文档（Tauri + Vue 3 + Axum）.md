# Todo List 桌面应用设计文档（Tauri + Vue 3 + Axum）

# 1. 项目概述

## 1.1 项目目标

开发一款支持 Windows、macOS、Linux 三端的轻量化桌面 Todo List 应用，核心实现任务增删改查、状态管理、系统级提醒功能；支持内置局域网 HTTP 服务，生成二维码供移动端浏览器扫码，实现移动端无 App 编辑 Todo 数据；预留桌面宠物扩展接口，后续可实现宠物与 Todo 任务的联动交互，打造便捷、个性化的任务管理工具。

## 1.2 项目范围

### 1.2.1 核心功能（当前迭代）

- Todo 任务管理：新增、编辑、删除、状态切换（待处理/已完成）、列表刷新。

- 系统通知提醒：支持为任务设置提醒时间，到达时间触发系统原生通知。

- 移动端扫码编辑：内置 Axum 局域网 HTTP 服务，生成二维码，移动端浏览器扫码后可直接增删改查 Todo 数据，与桌面端实时同步。

- 本地数据持久化：任务数据存储在本地，支持应用重启后数据不丢失。

- 基础 UI 适配：桌面端适配不同系统窗口风格，移动端适配手机屏幕尺寸。

### 1.2.2 扩展功能（后续迭代）

- 桌面宠物模块：创建悬浮式宠物窗口，支持简单动画、点击交互，与 Todo 任务联动（如任务完成时宠物反馈、提醒时宠物弹窗）。

- 数据备份与恢复：支持本地文件备份（JSON/SQLite）、云端备份（可选）。

- 任务分类与标签：支持任务按分类、标签筛选，提升管理效率。

- 自定义主题：支持明暗主题切换、自定义窗口颜色、字体大小等。

## 1.3 技术栈选型

|模块|核心技术|辅助工具/插件|选型说明|
|---|---|---|---|
|桌面框架|Tauri 1.5|tauri-plugin-store、tauri-plugin-notification|轻量化、跨三端、内存占用低，支持系统原生交互，完美兼容 Web 生态与 Rust 后端|
|前端渲染|Vue 3 + Vite|Element Plus、qrcode.vue、dayjs|开发效率高、热重载友好，Element Plus 提供完善 UI 组件，适配前后端一体化开发|

# 2. 需求分析

## 2.1 功能性需求

### 2.1.1 Todo 任务管理需求

- 新增任务：输入任务内容，支持快速提交，内容非空校验。

- 编辑任务：支持修改任务内容（后续扩展），切换任务状态（待处理/已完成）。

- 删除任务：支持单个任务删除，删除前可添加确认提示（可选）。

- 任务列表：展示所有任务，包含 ID、内容、状态，支持加载状态提示。

- 刷新列表：手动触发列表刷新，确保与本地数据同步。

### 2.1.2 提醒功能需求

- 提醒设置：为任务设置提醒时间（日期+时间），支持取消提醒。

- 系统通知：到达提醒时间时，触发系统原生通知（含任务内容），支持点击通知跳转至应用。

- 定时稳定性：应用最小化/后台运行时，提醒功能正常生效，不依赖前端页面。

### 2.1.3 移动端扫码编辑需求

- 二维码生成：生成包含局域网 HTTP 服务地址的二维码，实时更新服务地址（如 IP 变化）。

- 局域网服务：Axum 服务自动启动/停止（与应用生命周期一致），支持端口配置（默认 8080）。

- 移动端页面：扫码后跳转至适配手机的 Web 页面，支持任务增删改查，与桌面端实时同步。

- 连通性：手机与电脑需在同一 WiFi 下，支持自动识别局域网 IP，无需手动输入地址。

- 跨域支持：允许移动端浏览器访问 Axum 服务，避免跨域拦截。

### 2.1.4 数据存储需求

- 持久化：任务数据存储在本地文件，应用重启、电脑关机后数据不丢失。

- 同步性：桌面端、移动端的操作（增删改查）实时同步至本地存储，无延迟。

- 容错性：数据读写失败时，给出错误提示，避免应用崩溃，支持手动刷新恢复数据。

### 2.1.5 扩展功能预留需求

- 桌面宠物：预留悬浮窗口接口，支持无边框、透明、置顶、无任务栏图标配置。

- 联动逻辑：预留宠物与 Todo 任务的联动接口（如任务完成触发宠物动画）。

## 2.2 非功能性需求

- 跨端兼容性：支持 Windows 10+、macOS 12+、Linux（Ubuntu 20.04+、CentOS 8+），界面适配各系统风格。

- 性能需求：启动时间 ≤ 3s，列表加载 ≤ 500ms，API 响应 ≤ 300ms，内存占用 ≤ 100MB（优于 Electron 方案）。

- 易用性：UI 简洁直观，操作流程简单，无需复杂学习，扫码后可直接使用移动端功能。

- 轻量化：安装包体积 ≤ 10MB（三端通用），无多余依赖，无需额外安装运行环境。

- 稳定性：应用运行无崩溃、无卡顿，提醒功能不遗漏，数据不丢失、不错乱。

- 可扩展性：代码模块化设计，新增功能（如宠物、分类）时无需大规模修改核心代码。

## 2.3 约束条件

- 移动端：无需安装 App，仅支持主流手机浏览器（Chrome、Safari、微信浏览器、支付宝浏览器）。

- 开发环境：需安装 Node.js（v16+）、Rust（v1.70+）、pnpm/npm，确保 Tauri 编译环境正常。

- 局域网：移动端与电脑必须处于同一局域网，否则无法扫码连通（后续可扩展云端支持）。

- 权限：应用需获取系统通知权限、局域网访问权限，确保提醒功能与 HTTP 服务正常运行。

# 3. 总体设计

## 3.1 架构设计

应用采用「Tauri 主进程 + 前端渲染进程 + Axum 服务进程」的三层架构，基于 Rust 与 Web 技术栈的协同，实现跨三端、轻量化、高集成的功能，整体架构如下：

### 3.1.1 架构分层说明

1. 前端渲染进程（Vue 3 + Vite）：负责 UI 展示与用户交互，包括桌面端 Todo 界面、移动端 Web 界面、二维码生成，通过 HTTP 请求与 Axum 服务交互，通过 Tauri API 与主进程通信。

2. Tauri 主进程（Rust）：应用核心载体，负责系统集成（通知、窗口管理）、本地数据存储、Axum 服务启动/停止，提供全局状态管理，衔接前端与后端服务。

3. Axum 服务进程（Rust）：运行在 Tauri 主进程的后台线程，提供 RESTful API 接口、托管移动端静态页面，处理移动端与桌面端的数据同步请求，基于 Tokio 实现异步高效调度。

### 3.1.2 核心交互流程

- 应用启动流程：启动 Tauri 主进程 → 加载本地 Todo 数据 → 启动 Axum 服务 → 获取局域网 IP 生成二维码 → 启动前端渲染进程 → 展示桌面端界面。

- Todo 操作流程（桌面端）：用户操作（新增/编辑/删除）→ 前端发送 HTTP 请求 → Axum 服务处理请求 → 主进程更新本地存储 → 前端刷新列表。

- Todo 操作流程（移动端）：扫码打开 Web 页面 → 用户操作 → 移动端发送 HTTP 请求 → Axum 服务处理请求 → 主进程更新本地存储 → 桌面端/移动端同步刷新。

- 提醒功能流程：用户设置提醒时间 → 主进程记录提醒任务 → Tokio 定时调度 → 到达时间触发系统通知 → 点击通知跳转至应用。

- 应用关闭流程：停止 Axum 服务 → 保存当前 Todo 数据 → 关闭前端渲染进程 → 退出 Tauri 主进程。

## 3.2 模块划分

应用按功能划分为 5 个核心模块，各模块独立封装、低耦合，便于开发维护与扩展，模块划分如下：

|模块名称|所属进程|核心职责|关联模块|
|---|---|---|---|
|前端 UI 模块|前端渲染进程|桌面端/移动端 UI 展示、用户交互、二维码生成、请求发送与响应处理|Axum 服务模块、数据存储模块|
|Axum 服务模块|Tauri 主进程（后台线程）|RESTful API 实现、移动端页面托管、跨域处理、局域网服务启动/停止|前端 UI 模块、数据存储模块|
|数据存储模块|Tauri 主进程|本地数据读写、数据同步、容错处理，支持 Tauri Store 与 SQLite 切换|Axum 服务模块、提醒模块|
|提醒模块|Tauri 主进程|提醒时间管理、Tokio 定时调度、系统通知触发、通知跳转处理|数据存储模块、前端 UI 模块|
|扩展预留模块|Tauri 主进程 + 前端渲染进程|桌面宠物窗口管理、动画渲染、宠物与 Todo 联动接口预留|所有核心模块|
## 3.3 技术架构图（文字描述）

「用户」→ 桌面端 UI（Vue 3 + Element Plus）/ 移动端 Web UI（Vue 3 响应式） → 前端请求层（HTTP/HTTPS） → Axum 服务（API 路由 + 静态资源托管） → 全局状态（Arc+Mutex） → 数据存储模块（Tauri Store/SQLite）/ 提醒模块（Tokio 定时 + 系统通知）；同时，Tauri 主进程提供窗口管理、系统集成能力，扩展预留模块基于主进程 Window API 与前端 Canvas 实现桌面宠物功能。

# 4. 详细设计

## 4.1 前端 UI 模块详细设计

### 4.1.1 页面结构设计

#### 4.1.1.1 桌面端页面（App.vue 为主页面）

- 顶部导航栏：应用名称、主题切换（预留）、设置入口（预留）。

- 主体内容区（分两栏布局）：
        

    - 左侧（占比 2/3）：Todo 任务管理区，包含新增任务表单、任务列表、加载提示。

    - 右侧（占比 1/3）：二维码展示区、访问说明、局域网地址显示。

- 底部状态栏（可选）：应用版本、数据同步状态、Axum 服务运行状态。

#### 4.1.1.2 移动端页面（MobileTodo.vue 为核心组件）

- 顶部标题栏：应用名称、返回按钮（适配手机导航）。

- 主体内容区：新增任务输入框、任务列表（卡片式）、操作按钮（标记完成/删除）。

- 底部说明：简单提示（与电脑同一 WiFi、数据实时同步）。

### 4.1.2 组件设计

|组件名称|所属页面|核心功能|props/参数|
|---|---|---|---|
|TodoForm.vue|桌面端主页面|新增任务表单，内容输入、提交、非空校验|onSubmit（提交回调）|
|TodoList.vue|桌面端主页面|展示任务列表，状态切换、删除操作、加载提示|todoList（任务列表）、loading（加载状态）、onToggle（状态切换回调）、onDelete（删除回调）|
|QrcodeDisplay.vue|桌面端主页面|生成并展示二维码，显示局域网访问地址|url（二维码内容）|
|MobileTodoForm.vue|移动端页面|移动端新增任务，适配手机输入体验|onSubmit（提交回调）|
|MobileTodoItem.vue|移动端页面|移动端单个任务卡片，操作按钮适配手机尺寸|todo（单个任务）、onToggle（状态切换回调）、onDelete（删除回调）|
|ReminderForm.vue（预留）|桌面端主页面/TodoList|设置任务提醒时间，日期时间选择、取消提醒|todoId（任务ID）、onSetReminder（设置回调）、onCancel（取消回调）|
### 4.1.3 状态管理设计

当前迭代采用 Vue 3 内置的 reactive/ref 进行状态管理（轻量高效），复杂场景可扩展为 Pinia，核心状态如下：

- 桌面端状态：
        

    - newTodo：响应式对象，存储新增任务的内容、提醒时间（预留）。

    - todoList：响应式数组，存储当前展示的任务列表。

    - loading：布尔值，标记列表加载/请求处理状态。

    - qrcodeUrl：字符串，存储二维码对应的局域网访问地址（由主进程推送）。

- 移动端状态：
        

    - newTodoContent：字符串，存储新增任务内容。

    - todoList：响应式数组，存储移动端展示的任务列表。

    - apiBaseUrl：字符串，存储 Axum 服务地址（自动获取当前页面 origin）。

### 4.1.4 核心逻辑实现

#### 4.1.4.1 二维码生成逻辑

- 依赖 qrcode.vue 组件，传入 qrcodeUrl（局域网地址 + /mobile）生成二维码。

- 监听 Tauri 主进程发送的「web-server-url」事件，实时更新 qrcodeUrl，确保 IP 变化后二维码同步更新。

- 二维码级别设置为 H（最高容错级别），避免扫码失败。

#### 4.1.4.2 HTTP 请求封装

前端所有请求统一封装为工具函数（request.js），处理请求头、响应拦截、错误提示，核心请求如下：

- GET /api/todo：获取所有任务，用于列表刷新。

- POST /api/todo：新增任务，传入 content、status 参数。

- PATCH /api/todo/:id：更新任务状态，传入 status 参数。

- DELETE /api/todo/:id：删除指定任务，传入任务 ID。

错误处理：请求失败时，通过 Element Plus 的 ElMessage 给出提示（如网络异常、任务不存在），避免应用卡死。

#### 4.1.4.3 响应式适配逻辑

- 桌面端：采用 Element Plus 的栅格系统（el-row + el-col）实现两栏布局，适配不同窗口大小。

- 移动端：采用 Flex/Grid 布局，使用 vw/vh 单位设置组件尺寸，避免固定像素导致的适配问题。

- 移动端页面仅在扫码后展示，桌面端通过 v-show="false" 隐藏，避免占用资源。

## 4.2 Axum 服务模块详细设计

### 4.2.1 路由设计

Axum 服务采用 Router 分层路由设计，清晰划分 API 路由与静态资源路由，核心路由如下：

- API 路由：/api/todo 下的 GET/POST/PATCH/DELETE 接口，对应 Todo 增删改查操作。

- 静态资源路由：/mobile 路由，托管移动端 Web 页面（打包后的 MobileTodo 组件）。

- 根路由（可选）：跳转至桌面端前端页面，便于调试。

### 4.2.2 API 接口详细设计

|接口路径|请求方法|请求参数（JSON）|响应参数（JSON）|响应状态码|核心逻辑|
|---|---|---|---|---|---|
|/api/todo|GET|无|[{"id": "xxx", "content": "xxx", "status": "pending/completed"}]|200 OK|读取全局状态中的任务列表，返回数组|
|/api/todo|POST|{content: string, status: string}|{id: "xxx", "content": "xxx", "status": "xxx"}|201 Created|生成 UUID 作为任务 ID，新增任务至全局状态，同步更新本地存储|
|/api/todo/:id|PATCH|{status: string}|{id: "xxx", "content": "xxx", "status": "xxx"}|200 OK / 404 Not Found|根据 ID 查找任务，更新状态，同步更新本地存储；未找到则返回 404|
|/api/todo/:id|DELETE|无|字符串（删除成功/失败）|200 OK / 404 Not Found|根据 ID 删除任务，同步更新本地存储；未找到则返回 404|
### 4.2.3 跨域处理

依赖 tower-http 的 CorsLayer 实现跨域支持，配置如下：

- 允许所有来源（Origin: *），适配移动端不同浏览器的跨域请求。

- 允许所有请求方法（GET/POST/PATCH/DELETE）。

- 允许所有请求头，避免因自定义头导致的拦截。

### 4.2.4 服务启动与停止逻辑

- 启动时机：Tauri 主进程 setup 阶段，开启新的后台线程，启动 Axum 服务。

- 端口配置：默认使用 8080 端口，若端口被占用，可扩展为自动切换端口（后续迭代）。

- 局域网 IP 获取：调用 local-ip-address 库获取本地局域网 IP，若获取失败，默认使用 127.0.0.1。

- 服务地址推送：启动成功后，将局域网地址（http://ip:port）通过 Tauri 事件发送给前端，用于生成二维码。

- 停止时机：应用退出时，自动停止 Axum 服务，释放端口资源。

## 4.3 数据存储模块详细设计

### 4.3.1 存储方案选型与切换

#### 当前方案：Tauri Store

- 存储形式：键值对存储，数据以 JSON 格式保存到本地文件（todo_store.json）。

- 核心键值：以 "todos" 为键，存储任务列表的 JSON 字符串（HashMap<String, Todo> 序列化后）。

- 优势：上手简单、无需额外依赖，适合初期轻量数据存储，与 Tauri 生态完美兼容。

#### 扩展方案：SQLite（后续迭代）

- 依赖：tauri-plugin-sql，实现 SQLite 数据库的读写操作。

- 表设计：创建 todos 表，字段包括 id（主键，UUID）、content（文本）、status（文本）、reminder_time（文本，可选）。

- 切换逻辑：通过配置开关实现 Tauri Store 与 SQLite 的无缝切换，无需修改核心业务逻辑。

### 4.3.2 核心函数设计

#### 4.3.2.1 数据加载函数（load_todos_from_store）

- 参数：Tauri 窗口对象（用于获取 Store 实例）。

- 逻辑：获取 Store 中 "todos" 对应的 JSON 数据，反序列化为 HashMap<String, Todo>；若获取失败或反序列化失败，返回空 HashMap。

#### 4.3.2.2 数据保存函数（save_todos_to_store）

- 参数：Tauri 窗口对象、任务列表 HashMap。

- 逻辑：将任务列表序列化为 JSON 字符串，插入到 Store 的 "todos" 键中，调用 save() 方法保存到本地文件，确保数据持久化。

#### 4.3.2.3 容错处理

- 序列化/反序列化失败：返回默认空数据，避免应用崩溃，同时在控制台打印错误日志。

- 文件读写失败：提示用户数据保存失败，建议重启应用，同时尝试备份当前数据（可选）。

### 4.3.3 数据同步逻辑
